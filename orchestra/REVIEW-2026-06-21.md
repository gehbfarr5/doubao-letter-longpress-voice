## 1. 项目概况
LSPosed 模块：长按豆包输入法字母键触发语音输入，按住录音 / 原地松手上屏 / 滑到工具栏发送-换行 / 上下滑出取消。

- **代码体量**：4 个 Java 源文件，约 2570 行。
  - `DoubaoLetterLongPressHook.java`（2020 行）— 核心，5 个 Hook。
  - `DoubaoVoiceSendA11yService.java`（482 行）— 跨应用发送无障碍服务。
  - `BootRestoreReceiver.java`（38 行）、`XposedEntry.java`（28 行）。
- **当前版本**：versionName `1.5.0` / versionCode `6`（内部 Batch 1–6 对应 v1.5.1–v1.5.6）。
- **目标环境**：豆包 v1.3.11，Android 6.0+（API 23），LSPosed xposedminversion=82，compileSdk/targetSdk 36。

## 2. 总体评价：成熟、工程化程度高 ⭐⭐⭐⭐
这是一个迭代了 6 个主版本、文档化极好的逆向项目，不是玩具脚本：

- **防御性极强**：所有反射访问都有 try-catch，失败只退化为"功能不可用"而非崩溃；多层门槛（输入类型 / kbdType 黑名单 / 浮动单手模式 / 几何分区 / 滑动阈值）。
- **关键避坑沉淀清晰**：懒加载 Doubao 单例避免 `<clinit>` 过早触发整进程挂掉 —— 这是高价值的逆向经验，已写进类注释。
- **演进路线踏实**：v1.3.0 从"调魔数延迟"转向"事件驱动 callback"（`subscribeAsrAllBackThen` 订阅 `L$a` all-back 监听器），poll 仅作 fallback —— 体现了正确的工程判断。
- **跨密度/分辨率自适应**：几何判定基于 ratio，滑动阈值用 dp 转 px，平板手机通吃。
- **文档资产丰富**：`orchestra/` 下有 DOUBAO-INTERNALS、ADAPT-PLAYBOOK、各版本 PLAN/verify-report、FOLLOW-UPS，知识沉淀做得很好。

## 3. 发现的问题（按严重度）

### 🟡 中 — 文档与代码不一致
- **README 取消窗口时长自相矛盾**：
  - 代码实为 `CANCEL_WINDOW_MS = 500L`（README 第 28 行"0.5s"正确）。
  - 但 README 第 72 行使用表仍写"抑制所有 ASR commit **1.2 s**"，是 v1.5.3（1200→500ms）改动遗留的旧描述。
  - 类注释 head 也仍写"open a **1.2 s** suppression window"（第 79 行附近）。
  - **建议**：统一改为 0.5s。

### 🟢 低 — 死代码 / 未用占位
- `stopVoiceCommitFallback(...)`、`readCurrentEditboxAction(...)` 已无调用方（仅定义存在），是历史迭代遗留。`forceVad`、`pollAsrSettleAndEnter` 也基本被 `subscribeAsrAllBackThen` 取代，建议确认后清理（Batch 1 清过一轮，可再扫一遍）。
- `A11yService.PACKAGE_SEND_VIEW_ID` 全部 value=null —— per-package resource-id 精确匹配的"快路径"目前是空占位，从未命中，全靠启发式。属于待真机回填的预留结构，非缺陷，但值得在注释里更明确标注"尚未启用"。

### 🟢 低 — 版本号语义
versionName `1.5.0` 但实际包含 Batch 1–6 的全部改动，对外发布号与内部批次号（1.5.1–1.5.6）错位，容易让使用者误以为停在 1.5.0。建议发布时对齐到 `1.5.6` 或在 README/Release note 说明 1.5.x 含义。

### ℹ️ 设计权衡（非问题，记录在案）
- **a11y 发送选择器靠关键词启发式**（send/发送 + 排除词 + 优先级排序），App 大改版后节点结构变化会失效 —— 已有 logcat dump 兜底，属可接受的现实约束。
- **Gemini / Grok / Kimi 已预注册但未真机验证**（README 已诚实标注），是已知的"待验证"状态。

## 5. 下一步开发方向（建议优先级）

### A. 收尾真机验证（最高 ROI，低成本）
1. **真机验证 Gemini / Grok / Kimi 三个已预注册 App** 的 a11y 发送，按 logcat dump 回填选择器 / `PACKAGE_SEND_VIEW_ID`。这是当前唯一"已写代码但未确认能用"的部分，闭环成本最低。
2. **修文档不一致**（取消窗口 0.5s、版本号语义）—— 几分钟的事。

### B. 健壮性 / 可维护性
3. **per-package resource-id 精确匹配真正启用**：把 `PACKAGE_SEND_VIEW_ID` 从 null 占位填成实测值，启发式降为兜底，降低 App 改版导致误击的概率。
4. **死代码清理 + 一轮 lint**（Batch 1 的延续）。
5. **豆包多版本适配探针**：当前仅 v1.3.11。可加一个启动时的"混淆字段自检"日志（哪些 Hook 点解析成功/失败），让新版本失效时能快速定位，而不是静默退化。

### C. 功能演进（更大投入）
6. **Batch 7 / LLMCandidate（#13）**：逆向 `performLLMRequest` 参数语义，把语音结果接到豆包 LLM 候选窗做语义改写（不只是 ASR 整理）。这是 FOLLOW-UPS 里明确标注"需独立排期"的大项，价值高但逆向成本高。
7. **横屏 / 浮动键盘适配**：目前主动放弃。若有需求，需要重新设计非固定比例下的几何判定。
8. **扩展更多聊天类 App**（微信/Telegram 变体/各 AI App）：复用现有 a11y + force-send 双路基础设施，边际成本低，靠 ADAPT-PLAYBOOK 流程化推进。

### 建议的近期路线
**先做 A（真机收尾 + 修文档）→ 再做 B3（resource-id 精确匹配）**，把"已投入但未闭环"的部分先收干净；C 类按你对 LLM 候选窗的兴趣决定是否启动 Batch 7。
