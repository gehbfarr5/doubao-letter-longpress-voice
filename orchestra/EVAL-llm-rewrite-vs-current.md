# 评估：LLM 语义改写 vs. 现状 ASR 整理

> 日期：2026-06-21 ｜ 类型：功能演进评估（不含实现）｜ 对应 FOLLOW-UPS #13 / Batch 7

## 1. 两者各自是什么

### 现状：豆包 ASR 引擎「整理」
当前 commit 路径走 `AsrManager.q0()` / `p0(false,"")`，让豆包 ASR 引擎自然 finalize。它做的后处理是**转写级（transcription-level）**：
- 自动加标点
- 同音字纠正（"在"/"再"、"的"/"得"）
- 尾音补全、去口水词

**本质**：忠实于你说的话，只修"听写错误"，不改语义、不改措辞。

### 演进项：LLM 语义改写（`LLMCandidate` / `performLLMRequest`）
豆包输入法自带的 AI 改写能力，把已输入文字送给 LLM 做**语义级（semantic-level）**加工：
- 润色、正式化、口语→书面
- 扩写 / 缩写 / 改语气
- 语法修正、翻译

**本质**：会改写你的措辞、结构甚至语气——产出"更好的表达"，但不一定是"你原话"。

## 2. 逐维度对比

| 维度 | 现状（ASR 整理） | LLM 语义改写 |
|---|---|---|
| 加工层级 | 转写级（修错字/标点） | 语义级（改措辞/结构/语气） |
| 延迟 | 亚秒级 ~1s，本地/ASR 服务 | 1–3s+，依赖网络 LLM 往返 |
| 上屏即时性 | 松手即上屏，一步到位 | 需额外一步（生成候选→选用），或自动替换但显著拖慢上屏 |
| 确定性/可预测 | 高——所见即所说 | 低——可能"说的不是这个意思"，需 review |
| 失败模式 | 局部：漏字/错字，一眼可见 | 全局：整句跑偏/丢信息/胡编，更隐蔽 |
| 适用场景 | IM 快发、聊天、跨应用发送 | 正式文书/邮件/扩写 |
| 与本模块定位契合 | 高——核心卖点就是"快+即时" | 低——引入延迟+review，和"快"冲突 |
| 实现/逆向成本 | 已实现 | 高：需逆向 `performLLMRequest` 的 prompt/mode/回调语义 |
| 外部依赖 | 无额外依赖 | 可能需豆包账号登录/付费/配额；跨应用第三方输入框能否弹 LLM 候选窗存疑 |

## 3. 使用体验上的具体差别

1. **节奏被打断**：本模块现在的体感是"长按→说→松手→字就在那了/已发送"。LLM 改写要么在中间插一个"等 LLM + 看候选 + 选"的步骤，要么自动替换但让上屏从 1s 拖到 3s+。对"一句话快速发出去"的高频场景，这是体感质变（变慢）。

2. **从"听写"变成"代笔"**：现状你能信任输出就是你说的话，扫一眼标点即可发送。LLM 改写后，你**必须重读**确认它没改歪你的意思——尤其在 Claude/ChatGPT 这类对 prompt 精确度敏感的场景，被悄悄改写措辞反而有害。

3. **失败更难察觉**：错字一眼能抓；LLM 把"帮我查下周二的会"改成"请协助确认下周二的会议安排"——没错，但不是你想要的，且你可能没注意就发出去了。

4. **价值场景错位**：LLM 改写的增量价值在"超出转写范围的语义加工"（正式化/扩写/翻译）。但这些恰恰是用户**愿意慢、愿意 review** 的场景，而本模块的强场景（聊天、快发）恰恰**不需要**语义改写。两者用户意图不重叠。

## 4. 评估结论

- **增量价值**：中——在"正式度/书面质量"维度有提升，但属 nice-to-have，不是当前痛点（ASR 整理已覆盖日常转写质量）。
- **代价**：高——延迟 + 不确定性 + 强制 review，直接冲击本模块"快速即时"的核心体验；逆向成本和外部依赖（账号/配额/跨应用可用性）都显著高于现有功能。
- **定位冲突**：本模块卖点是"任意字母键→秒级语音→即时上屏/跨应用发送"。LLM 改写的特性（慢、需确认）与这个定位天然相悖。

### 建议
1. **不要替换默认路径**。若要做，做成**显式可选的二次润色**：上屏后用户主动触发，或作为候选窗里的一个可选项，绝不挡在"松手即上屏"的主路径上。
2. **优先级低于 A 类真机收尾**（Gemini/Grok/Kimi 验证、resource-id 回填）。这些是"已投入未闭环"，ROI 明确；Batch 7 是"高成本探索性增量"。
3. 若主要诉求是"语音转写质量"，现状已经够用；若诉求是"AI 帮我把话说得更好"，那其实更接近豆包自家 AI 改写功能本身的使用场景，模块化收益有限。

**一句话**：LLM 语义改写能让输出"更漂亮"，但会让本模块"变慢、变不可预测"——与它最有价值的快发/跨应用场景背道而驰。作为可选增强可以考虑，作为默认路径不建议。

## 5. 复核：「整理中」动画是不是 LLM？（2026-06-21 反编译核实）

> 起因：当前交互复刻空格长按，松手发送时有「整理中」动画，怀疑它是否已经在调 LLM、与计划中的 LLM 功能重复。核实对象：`/tmp/doubao-jadx`（v1.3.11 反编译产物）。

### 结论：不是 LLM，是 ASR 二段结果（转写级）

1. **模块复刻的空格长按路径 `AsrLongPressView` 里 LLM 引用数 = 0。** 「整理中」动画是 `AsrManager` 停止录音后进入 `SpeechStatus.KStoping` loading 态，在**等 ASR all-back / 二段结果**（`AsrContext.T(1,true)` 流式完成、`T(2,true)` 二段完成）。二段结果是 ASR 自己的二次识别（更准的标点/纠错/顺滑），**转写级，不改语义，不碰 LLM**。

2. **speech 包里那几处 LLM 引用都不是语音「整理」：**
   - `AsrManager` → `LLMRequest.e(bugDesc)` 是 `doFeedback`（**bug 反馈**上报），与上屏无关。
   - `AsrContext` → `LLMRequest.h()` 只读 `region` 配置（埋点），与上屏无关。
   - `AsrEditorLayoutView`（点麦克风进的**完整语音面板**）里确实有 `llm_candidate` 行 —— 但模块走的是长按 overlay，**不用这个完整面板**。

3. **豆包真正的 LLM 功能** = `KeyboardJni.performLLMRequest` / `input/llm.a`(`LLMCandidate`)：
   - 由**组合串变化**触发（`CompChangeToRequestLLM`，即打拼音时按 composition string 请求），是**打字时**的预测/改写。
   - 仅在「full input mode」下工作（`updateCandidateList, not in full input mode` 会跳过）。
   - 以**可点选候选**形式出现在候选窗，**不是自动替换**。

### 对原评估的修正

- **核心区分（转写 vs 语义）成立，无根本偏差**：「整理」与「LLM 改写」是两层不同的东西，豆包自己也分开实现（完整语音面板里两者并存）。**两者不重复。**
- **修正一处高估**：原文说 LLM「需逆向 + 自建改写管线，成本高」。实际豆包**已实现并在跑** `performLLMRequest`/`LLMCandidate`，所以更接近「触发现有功能 + 把候选 surface 到长按流里」，比"从零造"可行。但仍有真实障碍：① LLM 候选靠 **composition string** 驱动、需 **full input mode**；② 本模块最强场景是**跨应用发送（Claude/ChatGPT 第三方输入框）**，那里根本不会弹豆包候选窗 —— **LLM 候选在模块最值钱的场景基本用不上。**
- **推荐被豆包自身设计反向印证**：豆包把 LLM 做成**可点选候选、非自动替换**，正好就是原评估给的建议（"做成显式可选、不替换默认上屏路径"）。建议不变，且更有底气。
